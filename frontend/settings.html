🔎 CRM-Lookups (Jira)

Direkte Nachschlage-Endpunkte für die Anbindung von Telefonanlage/CTI oder Ticketsystem: aus einer Telefonnummer die zugehörige CRM-Kundennummer bestimmen bzw. zu einer CRM-Kundennummer alle Tickets abrufen. Beide sind GET-Aufrufe und akzeptieren Benutzer-Token oder externen API-Key. Setzt einen aktiven, konfigurierten Jira-Skill voraus.

Beispiel 3 – CRM-Kundennummer zu einer Telefonnummer

Sucht die Telefonnummer im Jira-CRM-Objektschema (Insight/Assets, Objektschema CRM, Standard-Schema-ID 21) – konkret in den Telefon-Attributen der CRM-Objekte (Zentrale Rufnummer, Telefonnummer, Mobil Nummer, …). Der Objekt-Key ist die CRM-Kundennummer (CRM-xxxxxx), es werden also ausschließlich CRM-Einträge zurückgegeben.

curl -sk "https://DEIN-JARVIS-HOST/api/jira/phonenumber?phone=%2B4920562611" \
  -H "X-API-Key: DEIN_API_KEY"

Die Nummer +4920562611 wird zu reinen Ziffern-Varianten normalisiert (+ = %2B in der URL): 4920562611 (voll international), 020562611 (national mit 0), 20562611 (ohne Vorwahl). Diese werden – von der längsten (eindeutigsten) zur kürzesten – per IQL like gegen die Telefon-Attribute geprüft; die erste Variante mit Treffern gewinnt. Telefonnummern liegen im CRM-Schema international/zusammenhängend (z. B. +4920562611).

Die Treffer werden nach Objekttyp sortiert: Organisationen zuerst, dann Organisationen-Produktgruppen, dann Personen. Der erste Treffer (Feld crm) ist damit die Organisation selbst. Antwort (Namen hier anonymisiert):

{
  "ok": true,
  "phone": "+4920562611",
  "crm": "CRM-66648",
  "found": true,
  "matches": [
    { "key": "CRM-66648",  "name": "Muster Klinikum GmbH",   "type": "Organisationen" },
    { "key": "CRM-421832", "name": "Muster Produktgruppe",    "type": "Organisationen Produktgruppen" },
    { "key": "CRM-158352", "name": "Muster Ansprechpartner",  "type": "Personen" }
  ],
  "total": 8,
  "variant": "4920562611"
}

phone (Aliase phonenumber, number)stringPflichtDie Telefonnummer (beliebige Schreibweise, z. B. +49 151 123456, 0151/123456). In der URL muss + als %2B kodiert werden.

🔎 CRM lookups (Jira)

Direct lookup endpoints for integrating a phone system/CTI or ticket system: determine the CRM customer number for a phone number, or fetch all tickets for a CRM customer number. Both are GET calls and accept a user token or an external API key. Requires an active, configured Jira skill.

Example 3 – CRM customer number for a phone number

Searches the phone number in the Jira CRM object schema (Insight/Assets, object schema CRM, default schema ID 21) – specifically in the phone attributes of the CRM objects (Zentrale Rufnummer, Telefonnummer, Mobil Nummer, …). The object key is the CRM customer number (CRM-xxxxxx), so only CRM entries are returned.

curl -sk "https://YOUR-JARVIS-HOST/api/jira/phonenumber?phone=%2B4920562611" \
  -H "X-API-Key: YOUR_API_KEY"

🛡️ Sicherheitsschicht – technische Funktionsanalyse

Diese Sicherheitsschicht schützt den Agenten gegen Rechteausweitung, Datenabfluss und Prompt-Injection durch Netzwerk-/Domain-Benutzer. Leitprinzip: Zugriffskontrolle wird im Code am Tool-Dispatch erzwungen – nicht als Bitte im System-Prompt. Sie ist dadurch nicht per Prompt, Base64-Kodierung oder „gelernten Fakten" umgehbar.

1. Rollenmodell

  • Lokale Administratoren (ALLOWED_USERS = jarvis/root sowie in Sicherheit → LDAP → Administratoren freigeschaltete AD-Konten): voller Zugriff, keine Einschränkung.
  • Netzwerk-/Domain-Benutzer (LDAP, nicht privilegiert): unterliegen sämtlichen unten beschriebenen Beschränkungen. Erkennung in agent.py über _LOCAL_PRIVILEGED_USERS; die Prüfungen laufen in _execute_tool() pro Tool-Aufruf.

2. Dateisystem-Confinement (filesystem-Tool)

Umgesetzt in backend/sandbox.py → authorize_fs(action, path). Pfade werden vollständig aufgelöst (Symlinks), um Ausbrüche wie /tmp/link → /etc/shadow zu verhindern.

AktionErlaubt für Domain-Benutzer
write / append / mkdirnur /tmp und data/documents (WRITE_ROOTS)
read / list / existsnur Wissens-/Arbeitsverzeichnisse (READ_ROOTS: /tmp, /mnt/jarvis-kb, data/knowledge, data/documents)

Immer gesperrt (is_sensitive()): Secrets (.env, settings.json, memory.json, auth_state.json, *.key/.pem/.crt/.p12, id_rsa, .ssh/, certs/), System (/root, /etc/shadow, /etc/sudoers, /boot, /proc, /sys) und App-Interna (data/settings.json, data/logs, Audit-/Konversations-Logs).

3. Shell-Härtung (shell_execute)

Mehrstufige Prüfung in agent.py + sandbox.authorize_shell():

  • Systemverändernde Befehle blockiert (_LDAP_SHELL_FORBIDDEN): rm, chmod, chown, apt/pip/npm install, systemctl, reboot, dd, mkfs, useradd, passwd, tee …
  • Verschleierung blockiert (SHELL_OBFUSCATION): base64 -d, xxd -r, eval, source, Pipe in bash/sh/python/perl/…. Schließt die Base64-Umgehung.
  • Secret-/Root-Pfade blockiert (SHELL_SECRET_PATHS) – interpreterunabhängig (greift auch bei cat oder python open()).
  • Schreib-Umleitungen (>) nur nach /tmp.

4. OS-Sandbox (harte Grenze)

Optional, aktiviert über die Einstellung sandbox_shell_user. Ist sie gesetzt, führt backend/tools/shell.py Shell-Befehle von Domain-Benutzern via runuser -u <user> -- /bin/bash -c … als unprivilegierter Betriebssystem-Benutzer mit Arbeitsverzeichnis /tmp aus (wrap_sandboxed()). Damit erzwingt das Betriebssystem die Grenze – unabhängig davon, wie ein Befehl formuliert oder kodiert ist (Python, Base64, …). Provisionierung: unprivilegierten Systembenutzer anlegen und Secrets per Dateirechten (chmod 600, /root = 700) unzugänglich machen. Ohne diese Einstellung greift „nur" die Code-Härtung (Punkte 2–3).

5. Keine Rechteausweitung über Sub-Agenten

Beim Start eines Sub-Agenten (z. B. coding_agent, Agent-Orchestrator) werden Benutzeridentität und Internet-Freigabe vererbt. Ein Sub-Agent eines Domain-Benutzers unterliegt denselben Confinement-Regeln; er läuft nicht mit leerem/privilegiertem Kontext.

6. Angriffserkennung (Guard)

In backend/security_guard.py, ausgewertet pro Chat-/Support-/WhatsApp-Eingabe (inspect()):

  • Heuristik: Regex-Signaturen bekannter Jailbreak-/Injection-Muster („ignore all previous instructions", DAN, „reveal system prompt", „bypass safety" …).
  • LLM-Klassifikator (optional): bewertet Eingaben zusätzlich. Hybrid (beide aktiv): der Klassifikator bestätigt jeden Heuristik-Verdacht vor der Sperre (weniger Fehlalarme).
  • Base64-Erkennung (decode_and_scan()): dekodiert Base64-Blöcke der Eingabe und prüft den Klartext auf Jailbreak-/Shell-/Secret-Indikatoren – schließt die Guard-Umgehung durch Kodierung.

7. Verstoß-Protokoll & automatische Sperre

Jeder im Dispatch verweigerte Zugriff wird als Verstoß protokolliert (record_violation()) – mit Tool, Klartext-Detail, auslösender Anfrage (Prompt), IP und Kanal. Ab einer Schwelle innerhalb eines Zeitfensters wird das Konto automatisch gesperrt.

  • Standard: 3 Verstöße in 600 s → Sperre. Konfigurierbar über security_autoblock_enabled, security_autoblock_count, security_autoblock_window.
  • Lokale Administratoren werden nie automatisch gesperrt (exempt).
  • Gesperrte Konten können sich noch anmelden, sehen aber nur den Sperr-Bildschirm samt Protokoll. Freischalten ausschließlich durch einen lokalen Benutzer (Gesperrte Konten → Freischalten).
  • Anzeige: „Letzte Zugriffs-Verstöße" und „Gesperrte Konten" in diesem Bereich; API: /api/security/violations, /api/security/incidents, /api/security/sandbox.
  • Internet-Egress-Sperre (harte Firewall-Grenze für Benutzer ohne Internet-Freigabe): Status via GET /api/security/egress (mit ?live=1 Live-Test), Ein-Klick-Einrichtung via POST /api/security/egress/setup, Deaktivierung via POST /api/security/egress/teardown. Bedienung unter Sicherheit → Berechtigungen → Harte Egress-Sperre.
  • OS-Sandbox (Systemschutz Netzwerk-Benutzer): Status via GET /api/security/sandbox (mit ?live=1 Isolationstest), Ein-/Ausschalten via POST /api/security/sandbox/setup bzw. /teardown. Bedienung unter Sicherheit → Berechtigungen → Systemschutz / Sandbox.
  • Root-Broker (Trennung UI-/Ausführungsebene): Betriebsmodus via GET /api/broker/status, Ein-Klick-Umschaltung via POST /api/broker/setup (getrennter Betrieb) bzw. POST /api/broker/teardown (Alt-Betrieb), Freigabeliste via GET /api/broker/ops, Admin-Entscheidung via POST /api/broker/ops/decide (Body {key, decision: allow|deny|pending}), Eintrag löschen via POST /api/broker/ops/remove, Audit-Log via GET /api/broker/audit?n=100. Bedienung unter Sicherheit → Root-Freigaben (Root-Broker).

8. Schutz vor Kontext-Vergiftung

  • Lern-Filter (learning.py → _sanitize_learned()): sicherheits-/rechte-bezogene „Fakten" (z. B. „root ist erlaubt", Secrets, sudo, base64) werden vor dem Speichern verworfen – es lässt sich keine Rechteausweitung „antrainieren".
  • Untrusted-Rahmung: Gedächtnis/gelernte Fakten werden dem Modell als [UNTRUSTED_CONTEXT] übergeben – reine Information, keine Anweisungen.
  • Unveränderliche Sicherheits-Grundregel an oberster Stelle des System-Prompts: Rechte kommen nur vom System; keine kodierten Anweisungen ausführen; Domain-Benutzer sind keine Administratoren. Diese Regel ist dem Modell übergeordnet, ersetzt aber nicht die Code-Erzwingung.

9. Konfiguration

  • Einstellungen → Sicherheit: Sicherheitsschicht (Master), Heuristik, LLM-Klassifikator; Anzeige Sandbox-Status + Verstöße + gesperrte Konten.
  • sandbox_shell_user – OS-Sandbox-Benutzer (leer = OS-Sandbox aus, nur Code-Härtung).
  • security_autoblock_* – Auto-Sperre-Schwelle/-Fenster.

10. Trennung UI-/Ausführungsebene (Root-Broker)

Im getrennten Betrieb läuft das Backend (Web-Oberfläche, Chat, Agent) als unprivilegierter Dienst-Benutzer; alles, was root braucht, läuft über den separaten Root-Broker (jarvis-broker.service, Unix-Socket). Der Broker kennt nur benannte, validierte Operationen (Dienst-Neustart, VNC, Sandbox, Mounts, Zertifikate …). Jede Operation wird beim ersten Auftauchen als auditierbarer Eintrag registriert: Systemoperationen automatisch erlaubt (vom Admin widerrufbar), beliebige Root-Shell-Befehle des Agenten dagegen erst nach expliziter Admin-Freigabe (Status „wartet auf Freigabe"). Policy und Audit-Log gehören root und sind vom Backend nicht manipulierbar. Verwaltung unter Sicherheit → Root-Freigaben.

11. Grenzen & Restrisiko

Die Muster-Prüfungen (Regex, Guard-Heuristik) sind Defense-in-Depth. Die harte Garantie für Shell-Befehle liefert die OS-Sandbox (Punkt 4) bzw. im getrennten Betrieb der unprivilegierte Dienst-Benutzer + Root-Broker (Punkt 10). Auf Alt-Installationen ohne Trennung (Backend als root) bleibt ein prinzipielles Restrisiko, da als root ausgeführter Code Wege finden kann. Empfehlung: Rechte-Trennung einrichten (deploy/security/setup_broker.sh), OS-Sandbox aktivieren und Secrets per Dateirechten absichern.

🧱 Harte Egress-Sperre – technische Funktionsanalyse

Die Egress-Sperre stellt sicher, dass Benutzer ohne Internet-Freigabe wirklich keine Informationen aus dem Internet beziehen oder Daten dorthin abfließen lassen können – auch nicht über Umwege. Leitprinzip: Die weiche Berechtigung (wer darf ins Internet) wird durch eine harte Grenze auf Betriebssystem-Ebene ergänzt, die unabhängig von der Formulierung eines Befehls wirkt.

1. Zwei Ebenen

  • Tool-Ebene (im Hauptprozess): Tools, die Informationen aus dem Internet holen (search_image, browser_control, browser_cdp, google_calendar/drive/gmail sowie jedes Tool mit requires_internet = True), werden im Dispatch (agent.py → _execute_tool) für nicht freigeschaltete Benutzer blockiert. Jira/Confluence sind intern (self-hosted) und daher bewusst nicht gesperrt.
  • Netzwerk-Ebene (harte Grenze): shell_execute solcher Benutzer läuft als separater, per Firewall netzwerkgesperrter OS-Benutzer. Das ist der Gegenstand dieser Beschreibung.

2. Warum eine zweite Ebene nötig ist

Die vorgelagerte Egress-Heuristik (_shell_hits_internet()) erkennt gängige Wege (curl, wget, ssh, git, urllib/requests/httpx, socket.connect …) und blockt sie mit klarer Meldung. Eine Regex kann ausgehenden Verkehr aber nicht lückenlos verhindern (z. B. rohe Sockets über eine Variable, eigene Protokolle). Diese Restlücke schließt die OS-Firewall.

3. Funktionsweise der Firewall

Umgesetzt in backend/egress_guard.py. Für Benutzer ohne Internet-Freigabe wird shell_execute via runuser als dedizierter OS-Benutzer jarvis_sandbox_noinet ausgeführt (Einstellung sandbox_shell_user_noinet). Für dessen UID gilt eine eigenständige nftables-Regel (table inet jarvis_egress, meta skuid-Match):

ZielRegel für den gesperrten Benutzer
Loopback (127.0.0.0/8, ::1)erlaubt
Internes LAN (10/8, 172.16/12, 192.168/16, 169.254/16)erlaubt – z. B. internes Jira/Confluence
DNS (Port 53) zu den konfigurierten Resolvernerlaubt (Namensauflösung)
Alles Übrige (öffentliches Internet)verworfen (DROP)

Die Tabelle ist eigenständig und beeinflusst bestehende Regeln (Docker/NAT) nicht. Ein systemd-Dienst (jarvis-egress.service) lädt sie nach jedem Reboot. UID und Resolver werden bei der Einrichtung automatisch erkannt – die Sperre funktioniert dadurch ohne manuelle Anpassung auf jedem Server.

4. Wer ist betroffen?

Nur Benutzer ohne Internet-Freigabe (Steuerung unter Sicherheit → Berechtigungen → Internet-Zugang via ad_internet_users/ad_internet_group). Lokale Administratoren und freigeschaltete Benutzer laufen über den normalen Sandbox-Benutzer mit Internet. Die beiden Ebenen (OS-Sandbox und Egress-Sperre) sind unabhängig schaltbar.

5. Einrichtung & Verifikation (in dieser Oberfläche)

  • Einrichten / Reparieren (POST /api/security/egress/setup): legt den OS-Benutzer an, schreibt die nftables-Regel + den Autostart-Dienst und setzt die Einstellung. Idempotent.
  • Status prüfen (Live-Test) (GET /api/security/egress?live=1): führt als gesperrter Benutzer einen echten Verbindungsversuch aus und zeigt, ob öffentliches Internet blockiert ist.
  • Deaktivieren (POST /api/security/egress/teardown): entfernt Regel + Autostart und leert die Einstellung. Die Tool-Ebenen-Sperre bleibt aktiv.

6. Grenzen & Restrisiko

  • DNS zu den internen Resolvern ist erlaubt (nötig für Namensauflösung); ein theoretisches DNS-Tunneling bleibt damit möglich. Für maximale Strenge lässt sich DNS für den gesperrten Benutzer ganz sperren.
  • Die Einstufung „intern" für Jira/Confluence gilt nur bei self-hostetem Betrieb. Zeigen sie auf eine Cloud (*.atlassian.net), wären sie „Internet" und müssten anders eingestuft werden.
  • In-Prozess-Tools (Bildersuche, Browser, Google) werden nicht über die Firewall, sondern über die Tool-Ebene gesperrt (Punkt 1) – die Firewall betrifft ausschließlich shell_execute.

🗜️ Wissens-Verdichtung – Funktionsweise

Problem: Das Lernsystem schreibt pro Konversation eine eigene Wissensdatei (data/knowledge/learned/JJJJ-MM/conv_<zeit>.md). Mit der Zeit sammeln sich viele Dateien mit doppelten oder widersprüchlichen Fakten an – jede Konversation lernt isoliert, niemand gleicht gegen den Bestand ab.

1. Was die Verdichtung macht

  • Sammelt alle Wissensdateien aus abgeschlossenen Monaten – der laufende Monat bleibt unangetastet.
  • Ein LLM-Lauf führt sie mit dem bisherigen konsolidierten Bestand zusammen: Duplikate werden zu einem Eintrag verschmolzen, bei Widersprüchen gewinnt der Fakt mit dem neueren Datum (das Datum steht im Kopf jeder Datei). Bei echter Unsicherheit bleiben beide Varianten mit Datum erhalten.
  • Das Ergebnis wird thematisch gruppiert in wenige Dateien geschrieben (learned/konsolidiert/), z. B. Server & Infrastruktur, Kunden, Vorgehensweisen.
  • Bei jedem weiteren Lauf werden neue Fakten gegen den Bestand abgeglichen – das Wissen konvergiert, statt jedes Mal neu geschrieben zu werden.

2. Sicherheit

  • Die LLM-Ausgabe läuft nochmal durch denselben Sicherheits-Filter wie das Lernen selbst (rechte-, passwort- und secret-bezogene Zeilen werden verworfen) – die Verdichtung kann den Prompt-Injection-Schutz nicht aushebeln.
  • Anweisungen, die innerhalb gelernter Fakten stehen, behandelt der Verdichter als Daten, nicht als Befehle.

3. Reversibel statt destruktiv

  • Originaldateien werden nicht gelöscht, sondern nach data/backups/learned_archiv/JJJJ-MM/ verschoben – von dort können sie jederzeit zurückgeholt werden.
  • Das Archiv liegt bewusst außerhalb des Wissensordners, damit die Suche es nicht erneut indexiert.
  • Der Vektor-Index (FAISS) wird direkt mitgepflegt: Originale raus, konsolidierte Themen-Dateien sofort rein – kein kompletter Neuaufbau nötig.
  • Alles-oder-nichts: Schlägt der LLM-Aufruf fehl, wird keine einzige Datei angefasst.

4. Auslösung

  • Manuell: Button „Verdichten" in diesem Panel. Die Anzeige daneben zeigt, wie viele Dateien beim nächsten Lauf verarbeitet würden.
  • Automatisch: Haken „monatlich automatisch" – ein Hintergrund-Dienst prüft dann regelmäßig, ob abgeschlossene Monate vorliegen, und verdichtet sie selbstständig (typisch: einmal pro Monatswechsel).
  • Die feedback_*.md-Dateien sind ausgenommen – Feedback ist Verhaltensregel, kein Faktenwissen, und wird nicht „wegverdichtet".

🔐 Root-Freigaben (Root-Broker) – technische Funktionsanalyse

Leitprinzip: Die Benutzeroberfläche (Chat, Support, Agent) und die Ausführungsebene mit Root-Rechten sind getrennte Prozesse. Das Backend läuft als unprivilegierter Dienst-Benutzer – selbst wenn ein Angreifer den Agenten per Prompt-Injection zu einem Befehl verleitet, läuft dieser nicht automatisch als root. Jede Root-Aktion ist eine benannte, geprüfte und protokollierte Operation.

1. Architektur

  • jarvis.service (User=jarvis): Web-UI, Chat, Agent, Skills. Bindet Port 443/80 über die Linux-Capability CAP_NET_BIND_SERVICE – ganz ohne root.
  • jarvis-broker.service (root): kleiner, separater Dienst (backend/broker/daemon.py). Übernimmt beim Start die root-pflichtigen Systemaufgaben (Firewall-Ports, VNC/x11vnc, Bildschirmsperre) und lauscht danach auf dem Unix-Socket /run/jarvis-broker.sock.
  • Der Socket gehört root:jarvis mit Rechten 0660 – nur der Dienst-Benutzer kann Anfragen stellen; Netzwerkzugriff auf den Broker gibt es nicht.

2. Benannte Operationen statt „root darf alles"

Der Broker kennt ausschließlich fest definierte Operationen mit harter Argument-Validierung (backend/broker/ops.py):

  • systemctl – nur Whitelist-Units (jarvis, whatsapp-bridge, lightdm, egress, broker) und erlaubte Aktionen.
  • unlock_screen, switch_session, vnc_restart – Desktop-/VNC-Steuerung.
  • chpasswd – Linux-Kennwort setzen (Erst-Login-Pflichtwechsel).
  • sandbox_exec – Shell-Befehle von Netzwerk-Benutzern als Sandbox-OS-User (runuser); nur Benutzer mit Prefix jarvis_sandbox, nie uid 0.
  • sandbox_*/egress_* – Ein-Klick-Einrichtung von OS-Sandbox und Egress-Firewall.
  • mount_share/umount_share – Wissens-Freigaben (SMB/NFS/WebDAV), ausschließlich unterhalb /mnt/.
  • certbot_obtain – Let's-Encrypt-Zertifikat inkl. Renewal-Hook.
  • shell_root – generischer Root-Shell-Befehl. Läuft niemals sofort, siehe Punkt 3.

3. Auditierbare Freigabeliste („wenn sie auftauchen")

Jede Operation wird beim ersten Auftauchen automatisch als Eintrag in der Freigabeliste registriert (/etc/jarvis/broker-policy.json, nur für root beschreibbar):

  • Systemoperationen (Punkt 2, außer shell_root): automatisch erlaubt, als „Systemoperation" markiert und jederzeit per Klick widerrufbar („Ablehnen").
  • shell_root:<befehl>: startet immer als „wartet auf Freigabe" (pending). Der Agent erhält eine klare Meldung, der Befehl wird nicht ausgeführt. Erst nach „Erlauben" durch einen lokalen Administrator ist genau dieser Befehl (exakte Signatur) ausführbar – „Ablehnen" sperrt ihn dauerhaft.
  • Jeder Eintrag zeigt: Beschreibung, Status, wer ihn angefordert hat, Anzahl/Zeitpunkt der Nutzung, wer entschieden hat. Löschen (×) setzt den Eintrag zurück – er erscheint beim nächsten Auftauchen erneut.

4. Ablauf eines Root-Befehls des Agenten

  • 1. Agent ruft shell_execute mit z. B. apt-get install … auf.
  • 2. shell.py::_needs_root() erkennt den Root-Bedarf (sudo, systemctl, apt, mount, useradd, …; reine Lesebefehle wie systemctl status laufen weiter unprivilegiert).
  • 3. Der Befehl geht als shell_root an den Broker → Freigabeliste: pending (neuer Eintrag) / erlaubt (Ausführung mit Live-Streaming) / abgelehnt (Verweigerung).
  • 4. Bei „pending" informiert der Agent den Benutzer, dass ein Administrator die Anfrage unter Sicherheit → Root-Freigaben freigeben muss.

5. Audit & Manipulationsschutz

  • Jede Anfrage – auch verweigerte – landet im Audit-Log /var/log/jarvis-broker-audit.jsonl (Zeitpunkt, Benutzer, Operation, Entscheidung, Exit-Code, Dauer). Einsehbar über „Audit-Log anzeigen".
  • Policy-Datei und Audit-Log gehören root (0600) – das unprivilegierte Backend kann sie nicht verändern oder löschen; Passwörter werden vor der Protokollierung maskiert.
  • Die Statusanzeige oben im Bereich zeigt immer den Betriebsmodus: 🟢 getrennter Betrieb, 🟠 Alt-Betrieb (Backend noch als root, Liste+Audit aktiv, keine Prozess-Trennung), 🔴 Broker nicht erreichbar.

6. Betriebsarten & Migration

  • Getrennt (empfohlen): Ein-Klick-Aktivierung unter Sicherheit → Root-Freigaben („Getrennten Betrieb aktivieren / reparieren") oder manuell mit bash deploy/security/setup_broker.sh (Eigentümerschaft, systemd-Units, Dienste, Verifikation). Beide Wege sind idempotent; die Dienste starten dabei neu.
  • Alt-Betrieb: Nicht migrierte Installationen funktionieren unverändert – der Broker-Client führt Operationen dann lokal aus, inklusive Freigabeliste und Audit. Rückweg per Klick („Deaktivieren (Alt-Betrieb)") oder bash deploy/security/teardown_broker.sh.
  • API (nur lokale Administratoren): GET /api/broker/status, POST /api/broker/setup, POST /api/broker/teardown, GET /api/broker/ops, POST /api/broker/ops/decide, POST /api/broker/ops/remove, GET /api/broker/audit.

7. Grenzen & Restrisiko

  • Freigabe-Entscheidungen kommen über die Web-UI zum Broker: Ein vollständig kompromittiertes Backend könnte sich theoretisch selbst Freigaben erteilen. Die Trennung schützt primär davor, dass Agent-/Prompt-Injection-Code direkt mit root läuft – und macht jede Root-Aktion sichtbar und nachvollziehbar.
  • Ein einmal erlaubter shell_root-Befehl darf erneut ausgeführt werden, bis er widerrufen wird. Empfehlung: eng gefasste Befehle freigeben, keine „Sammel-Freigaben" (z. B. kein bash -c … mit variablem Inhalt).
  • Die Root-Erkennung in der Shell ist eine Heuristik: Nicht erkannte Befehle laufen unprivilegiert und scheitern dann schlicht an fehlenden Rechten („Permission denied") – es findet keine stille Rechteausweitung statt.
